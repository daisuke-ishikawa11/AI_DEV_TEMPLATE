# Windows 安定動作ルール（作業前確認必須）

ターミナルで日本語や UTF-8 を扱う際、エンコーディング未設定は文字化け・再試行ループ・誤解析の原因になります。

**コマンド実行時は以下を必ず適用すること。**

| 環境 | 対策 |
|------|------|
| PowerShell | コマンド冒頭に `[Console]::OutputEncoding = [System.Text.Encoding]::UTF8;` |
| CMD | コマンド冒頭に `chcp 65001 > nul &&` |
| Python | PowerShell では `$env:PYTHONUTF8=1;` を前置 |
| Docker | `environment` に `PYTHONUTF8=1` を含める |
| ファイル書き込み | `Set-Content` 等で `-Encoding UTF8` を明示 |

ソースファイルの保存形式は **UTF-8（BOM なし推奨）** とする。

# 誠実性と事実確認の原則 (Honesty & Verification)

> [!CAUTION]
> 「ごまかし」「未確認の推測」「事実の隠蔽」は、技術的ミス以上に重大なルール違反とみなします。

1. **思い込みの禁止**  
   数値の 0 / N/A、テスト失敗、想定外の挙動に対し、「初期状態だから」「仕様だから」といった**推測のみ**で結論を出さない。

2. **一次ソース確認の義務**  
   回答・修正提案の前に、該当する **生ログ、テスト結果、再現手順、関連コード** を確認し、根拠を示す。  
   例: アプリログ、`tests/` の出力、CI の失敗ログ、再現用の最小コマンド。

3. **不明点の即時開示**  
   不具合の可能性や知識不足に気づいたら、ごまかさず報告し、調査・修正方針を提案する。

4. **説明より修正を優先**  
   安易な説明で責任を回避せず、原因特定と修正（または再現可能な Issue 化）を優先する。

5. **調査の徹底**  
   質問・指摘には、ログ確認・コード精読・検証コマンド実行などを行ってから回答する。調査なしの断定は避ける。

6. **ナレッジベース（brain）の参照**  
   本テンプレートを利用する場合、作業前および不明点があるときは `brain/` を確認する。  
   - 構造化ナレッジ: `brain/wiki/`  
   - 過去の調査・レポート: `brain/reports/`  
   - レガシー運用がある場合: `brain/MEMORY.md`, `brain/daily/`（**存在する場合のみ**。破壊・移動禁止）

# 推奨ルール：外科手術の原則 (Surgical Principles)

AI 支援開発におけるスコープクリープ（不要な修正によるデグレ）を防ぐための最優先ルールです。

## ルール1：切開範囲を事前に宣言する

修正前に必ず対象を明示する。

```markdown
【変更対象】[ファイル名] [クラス/関数名]
【変更範囲】[追加/修正内容の詳細]
【変更理由】[なぜこの変更が必要か]
```

## ルール2：既存コードへの変更は「追加のみ」を原則とする

- ✅ OK: 新しい関数・ファイルの追加、バグ修正に必要な最小限の変更
- ❌ NG: 無関係なリファクタ、フォーマット一括変更、「ついでに」の改善

## ルール3：1 タスク = 小さな差分

1 回の作業で変更範囲を広げすぎない。可能なら **1 ファイルずつ** 動作確認と完了報告を行う。

## ルール4：スコープ外は DEFER.md に記録する

本来の目的以外で気になった点は、その場で直さずリポジトリルートの `DEFER.md` に追記し、別タスク・Issue として扱う。

# ログ・外部データの扱い (Logging & Data Hygiene)

LLM 連携やログパイプラインを扱うプロジェクト向けの汎用ガイドです。

- **エンコーディング**: ログ出力・パース前に UTF-8 を前提とし、必要なら Unicode 正規化（NFC）と制御文字の除去を行う。
- **PII・秘密情報**: API キー、トークン、秘密鍵、個人情報をログ・コミット・レポートに含めない。マスクまたは環境変数で注入する。
- **再現性**: バグ報告には「いつ・何を実行したか・期待と実際」を残す。

# 日次報告・記録ルール (Daily Reporting Protocol)

本テンプレートの `brain/` を使う場合の推奨運用です。プロジェクトで別の SSOT を定めている場合はそちらを優先してください。

1. **レポートの保存先**  
   - 調査・まとめ: `brain/reports/`（Karpathy 方式）  
   - 既存の日次運用がある場合: `brain/daily/`（レガシー。**破壊・移動禁止**）

2. **ファイル命名**  
   - `brain/reports/`: `{YYYY-MM-DD}_{topic-slug}.md`  
   - `brain/daily/`: プロジェクト慣例に合わせる（例: `YYYY-MM-DD_final_report.md`）

3. **内容**  
   その日の実装・修正・調査結果・再現手順を 1 ファイルに集約し、結論を冒頭に置く。

4. **メモリ更新（任意）**  
   `brain/MEMORY.md` が存在する場合のみ、サマリーとレポートへのリンクを追記する。

5. **バージョン管理**  
   ユーザーが明示した場合、またはプロジェクト慣例で定めている場合に限り `git commit` / `push` を行う。秘密情報を含むファイルはコミットしない。

# ターミナル操作の補助：PowerShell 版 grep 代替 (sls)

Windows で `grep` が使えない場合、`Select-String`（エイリアス: `sls`）を使う。

| Linux/bash | PowerShell | 備考 |
|---|---|---|
| `grep "pattern" file` | `sls "pattern" file` | 基本検索 |
| `grep -r "pattern" dir` | `sls -Path "dir\**\*.py" -Recurse` | 拡張子指定の再帰検索 |
| `grep -n "pattern" file` | `sls "pattern" file` | 行番号付き |
| `cat file \| grep "pattern"` | `gc file \| sls "pattern"` | パイプライン |
| `grep -v "pattern"` | `sls "pattern" file \| Where-Object { $_ -notmatch "pattern" }` | 除外（用途に応じて調整） |
